# Feature Specification: MVI Pattern for UI-FBP Interface Generation

**Feature Branch**: `087-mvi-ui-fbp`
**Created**: 2026-05-03
**Status**: Draft

## Clarifications

### Session 2026-05-03

- Q: How committed are we to MVI's actual guarantees (SSOT, unidirectional flow, test isolation), given that retaining today's singleton State as a renamed `StateStore` would deliver the MVI surface but not the semantics? → A: **Design B — eliminate the singleton.** Per-port flows move onto the `DynamicPipelineController` instance; `ControllerInterface` gains additive `emit<SourcePort>(...)` methods; the Sink CodeNode receives a controller-injected "sink reporter" callback at runtime construction. The State data class becomes the genuine SSOT for the UI; multiple flow-graph instances are state-isolated; the source-port path goes through the controller (no shared mutable singleton). FR-007 is softened from "MUST keep their current signatures" to "MUST preserve existing methods; additive extensions OK."

**Input**: User description: "Use MVI pattern in viewmodel for UI-FBP interface. Modify the code generation for the UI-FBP interface generation to follow the MVI architecture pattern. UI screens take 2 parameters, a single state parameter and an onEvent parameter. For this feature, first migrate the DemoUI module to this pattern, and after that modify other DemoProjects to follow."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — UI-FBP generator emits MVI-shaped artifacts (Priority: P1)

A module author opens the UI-FBP generator on a UI module and runs code
generation. The output now follows the Model-View-Intent (MVI) pattern: a
single immutable State, a sealed Event hierarchy, and a ViewModel that
exposes state as an observable and dispatches events through `onEvent`. The
author's hand-written `{Name}Screen` composable then uses the canonical
two-parameter signature `(state, onEvent) -> Unit` and is wired to the
ViewModel by the hosting app's own root/navigation layer (which is not
generated by UI-FBP).

**Why this priority**: This is the load-bearing change — every UI module
generated after this feature lands depends on it. Without P1, the migration
work in P2/P3 cannot start.

**Independent Test**: Run the UI-FBP generator on a synthetic test spec
(matching the shape used by `UIFBPInterfaceGeneratorTest`) and assert the
output file set contains: a `{Name}State.kt` data class, a `{Name}Event.kt`
sealed interface, and a `{Name}ViewModel.kt` exposing
`state: StateFlow<{Name}State>` and `fun onEvent(event: {Name}Event)`. The
generated module compiles standalone, and a hand-written `{Name}Screen` with
signature `(state: {Name}State, onEvent: ({Name}Event) -> Unit) -> Unit`
compiles against it.

**Acceptance Scenarios**:

1. **Given** a UI-FBP spec with `sinkInputs = [foo: Int, bar: String?]` and
   `sourceOutputs = [click: Unit]`, **When** the generator runs, **Then** the
   emitted `{Name}State.kt` is a `data class` whose properties mirror
   `sinkInputs` (each with sensible defaults) AND the emitted `{Name}Event.kt`
   is a `sealed interface` with one case per `sourceOutput`.
2. **Given** the same spec, **When** the generated `{Name}ViewModel` is
   constructed and the author writes a `{Name}Screen` consuming it, **Then**
   the screen's signature is exactly `fun {Name}Screen(state: {Name}State,
   onEvent: ({Name}Event) -> Unit)` — the screen has no other parameters and
   no direct ViewModel dependency. The hosting app's own root composable (not
   generated by UI-FBP) is responsible for collecting `viewModel.state` and
   passing `viewModel::onEvent`.
3. **Given** generation succeeds, **When** the consuming UI file invokes
   `onEvent({Name}Event.SomeCase)`, **Then** the ViewModel's `onEvent` updates
   the in-memory state in a way observable through `state.value`.

### User Story 2 — DemoUI module migrated to the MVI shape (Priority: P2)

The author regenerates the DemoUI module's UI-FBP artifacts using the new
generator. The hand-written DemoUI screen file is updated once to use the
two-parameter `(state, onEvent)` signature, and DemoUI's own root/navigation
composable is updated to collect `viewModel.state` and pass `viewModel::onEvent`
to the Screen. The module compiles, the runtime preview opens via the
GraphEditor's Runtime Preview panel, and the user interactions exercised in
the original DemoUI work identically (button taps, displayed values, animation
behaviour) — no functional regression vs the pre-migration baseline.

**Why this priority**: Validates that the generator change works on the real
module that drove the pattern's design. Without P2, the generator change is
unproven on any production-shaped consumer.

**Independent Test**: Regenerate DemoUI's UI-FBP artifacts. Open Runtime
Preview from the GraphEditor with the DemoUI flow graph selected. Drive every
documented interaction in the DemoUI demo and verify the displayed state +
emitted side-effects match the pre-migration behaviour.

**Acceptance Scenarios**:

1. **Given** the new generator and a clean DemoUI working tree, **When** the
   author runs UI-FBP generation, **Then** every prior DemoUI generated file
   is replaced with the MVI-shaped equivalent and DemoUI compiles.
2. **Given** the migrated DemoUI module, **When** Runtime Preview is opened,
   **Then** the screen renders, and every interaction the user performed in
   the pre-migration version produces the same observable output.
3. **Given** the migrated DemoUI module, **When** the FBP runtime delivers a
   value on a sink-input port, **Then** the UI's displayed state reflects that
   value within one frame (no perceptible lag vs baseline).

---

### User Story 3 — Other DemoProject UI modules adopt the same shape (Priority: P3)

After P1 + P2 ship, the other DemoProject UI modules (StopWatch,
KMPMobileApp, WeatherForecast, and any other UI-bearing modules) can be
regenerated with zero generator-side changes. Each module's hand-written UI
file is updated to use the new two-parameter signature, and the module's
runtime preview behaves identically to its pre-migration baseline.

**Why this priority**: Confirms the generator change generalizes — any
per-module branching would invalidate the design. Lower priority than P2
because each module's migration is independent and can be staged over time.

**Independent Test**: For each DemoProject UI module, regenerate, compile,
open Runtime Preview, and verify the demo's golden-path interactions still
work. Treat each module's migration as a separate verification task.

**Acceptance Scenarios**:

1. **Given** the new generator (unchanged from P1) and any DemoProject UI
   module, **When** the author runs UI-FBP generation on that module, **Then**
   the output follows the same MVI shape as DemoUI's output — no module-specific
   special-casing in the generator.
2. **Given** a migrated DemoProject UI module, **When** Runtime Preview is
   opened, **Then** the demo behaves identically to its pre-migration baseline.

### Edge Cases

- **UI module with zero events** (display-only screen, no `sourceOutputs`):
  The generator MUST still emit a (possibly empty) `sealed interface
  {Name}Event` and the screen still takes `onEvent: ({Name}Event) -> Unit` so
  every screen has the same shape regardless of intent count. The empty
  sealed interface signals "no user-driven events" without requiring callers
  to special-case the signature.
- **UI module with zero state** (output-only, no `sinkInputs`): The State
  data class is still emitted (with no fields) so the screen still takes
  `state: {Name}State`. Symmetry with the empty-event case.
- **Sink-input port carrying a nullable type** (e.g., `Post?`): The State
  property is nullable with a `null` default. Matches the user's example
  `val postDetail: Post? = null`.
- **State property whose default cannot be derived** (custom IP type with no
  no-arg constructor): The generator emits the property as nullable with a
  `null` default, regardless of the source port's nullability declaration.
  Authors who need a non-null default override post-generation.
- **Mid-emission generator failure**: When generation fails for any reason,
  the partial output set MUST NOT replace the prior generation's files
  (atomicity — same as today).
- **Hand-written UI file that hasn't been updated to the new signature**:
  The compile fails loudly at the screen call site. No fallback shim — the
  migration is intentionally a one-time breaking change per module.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The UI-FBP code generator MUST emit a `data class
  {FlowGraph}State` whose properties mirror the spec's `sinkInputs` ports.
  Each property is declared `val`, has a default value derived from the
  port's declared type (`0` for `Int`, `false` for `Boolean`, `""` for
  `String`, `null` for nullable or unknown types), and is exposed as a
  constructor parameter so the data class is freely copyable.
- **FR-002**: The UI-FBP code generator MUST emit a `sealed interface
  {FlowGraph}Event` with exactly one case per spec `sourceOutput` port. Each
  case carries the port's value as its sole property. Naming convention:
  `data class Update<PortName>(val value: <PortType>) : {FlowGraph}Event` for
  ports carrying values, `data object <PortName>` for `Unit`-typed ports.
- **FR-003**: The UI-FBP code generator MUST emit a `class {FlowGraph}ViewModel`
  that exposes a `state: StateFlow<{FlowGraph}State>` (read-only public surface)
  and an `fun onEvent(event: {FlowGraph}Event)` dispatcher. The ViewModel
  internally holds a `MutableStateFlow<{FlowGraph}State>` and updates it via
  `copy(...)` calls in response to incoming sink-port values collected from
  the controller's per-sink-port `StateFlow`s; the `onEvent` dispatcher routes
  each event to the matching `controller.emit<SourcePort>(...)` method on the
  underlying `{FlowGraph}ControllerInterface`. The ViewModel is the sole owner
  of state — no shared mutable singleton intermediates the source/sink paths
  (per the 2026-05-03 clarification, Design B).
- **FR-004**: The UI-FBP generator MUST publish a contract that hand-written
  `{FlowGraph}Screen` composables take exactly two parameters: `state:
  {FlowGraph}State` and `onEvent: ({FlowGraph}Event) -> Unit`. The contract
  is published implicitly through the generated public types — `{FlowGraph}State`
  (data class), `{FlowGraph}Event` (sealed interface), and `{FlowGraph}ViewModel`
  (exposing `state: StateFlow<{FlowGraph}State>` + `fun onEvent({FlowGraph}Event)`).
  No separate publication artifact (README excerpt, KDoc page, or contract
  test) is required: any hand-written Screen with the canonical signature
  compiles against the generated trio, and any non-conforming Screen fails
  the host module's compile (FR-011's loud-failure surface). The Screen
  itself is NOT generated by UI-FBP — the hosting app's own root/navigation
  layer is responsible for collecting `viewModel.state` and passing
  `viewModel::onEvent` to a hand-written Screen with this signature.
- **FR-006**: The DemoUI module MUST be migrated to the new generator output
  in this feature. Migration includes regenerating UI-FBP artifacts and
  updating the hand-written DemoUI screen file to the two-parameter
  signature. Other DemoProjects (StopWatch, KMPMobileApp, WeatherForecast, …)
  follow per P3 — they are in-scope but lower-priority.
- **FR-007**: The MVI generator output MUST preserve the existing
  Runtime Preview contract additively. The `create{FlowGraph}Runtime(...)`
  factory function keeps its current signature so the Runtime Preview panel
  and `ModuleSessionFactory` keep working without changes. The generated
  `{FlowGraph}ControllerInterface` MUST preserve every method it exposes
  today (`start / stop / pause / resume / reset / getStatus / executionState`
  inherited from `ModuleController`, plus the per-sink-port `StateFlow<T>`
  members); it MAY gain additive `fun emit<SourcePort>(value: T)` methods
  per source-output port (per the 2026-05-03 clarification, Design B).
  Additive changes are non-breaking for existing consumers.
- **FR-008**: The generator MUST handle the empty-events case (zero
  `sourceOutputs`) by emitting an empty `sealed interface {FlowGraph}Event`
  and a no-op `onEvent` body. Screen callers still pass `onEvent` even
  though no event case can ever be constructed.
- **FR-009**: The generator MUST handle the empty-state case (zero
  `sinkInputs`) by emitting a `data class {FlowGraph}State()` (zero-arg
  data class) and a `state` flow that emits exactly one `{FlowGraph}State()`
  value.
- **FR-010**: When generation fails partway, no partial file set replaces
  the prior generation. Atomicity is preserved by writing through the same
  staged-replacement mechanism used today (`UIFBPSaveService` semantics).
- **FR-011**: The migration MUST NOT introduce a backwards-compatibility
  shim: the prior `viewModel.emit(...)` API AND the singleton-State
  `{FlowGraph}State` object are removed wholesale (the singleton is
  eliminated entirely under Design B — no rename-to-StateStore intermediate).
  Hand-written callers must update to the new signature once per module.
  (Rationale: a shim or surviving singleton would let stale call sites
  compile silently against the old API and mask half-migrated modules; a
  surviving singleton would also defeat the test-isolation guarantee that
  motivated Design B.)
- **FR-012**: The generator MUST emit deterministic, gitignore-friendly
  output: re-running generation on an unchanged spec produces byte-identical
  files (no timestamps, no file ordering instability).

### Key Entities

- **State (`{FlowGraph}State`)**: Immutable snapshot of every value the UI
  observes from the FBP graph. One property per sink-input port. Carried as
  a constructor-driven `data class`, so the ViewModel's
  `MutableStateFlow<{FlowGraph}State>` can update one property at a time via
  `copy(...)` and downstream observers receive a fresh snapshot per change.
- **Event (`{FlowGraph}Event`)**: Closed hierarchy of user intents the
  Screen can raise. One case per source-output port. Each case carries the
  value the corresponding port should emit when handled.
- **ViewModel (`{FlowGraph}ViewModel`)**: The bridge between the FBP
  controller and the Compose UI. Subscribes to the controller's typed
  observable surface and folds each emission into a new State snapshot;
  routes incoming events to the controller's source-port emit calls.
- **Screen (`{FlowGraph}Screen`)** *(hand-written, not generated)*: Pure
  two-parameter composable the author writes inside the host module.
  Receives a snapshot of State and an event-dispatch callback, renders, and
  is fully preview-friendly without any ViewModel context. UI-FBP only
  publishes the contract — the hosting app's root/navigation layer wires
  the ViewModel to this Screen.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of UI-FBP generator outputs after this feature lands
  expose a ViewModel + State + Event triple that supports the canonical
  Screen signature `(state: {Name}State, onEvent: ({Name}Event) -> Unit) ->
  Unit`. Because UI-FBP does NOT generate the Screen itself (FR-004), the
  contract is verified by exemplar: a hand-written test Screen with this
  signature compiles against every generated artifact set. Specifically,
  `UIFBPViewModelGeneratorTest` (T015) asserts the emitted ViewModel's
  public surface is exactly `state: StateFlow<{Name}State>` +
  `fun onEvent({Name}Event)`; the multi-instance isolation test (T033)
  exercises this contract end-to-end against the regenerated DemoUI module.
- **SC-002**: The DemoUI module, after migration, behaves identically to
  its pre-migration baseline along every documented interaction in its
  Runtime Preview demo. Verified by side-by-side runtime preview comparison.
- **SC-003**: Adding a new user intent to a migrated module (state field +
  event case + handling) takes ≤ 3 file edits (state property, event case,
  ViewModel `onEvent` branch). Verified by walk-through on DemoUI of one
  added intent post-migration.
- **SC-004**: At least one DemoProject UI module beyond DemoUI (P3 scope)
  is successfully migrated using the same generator with zero generator-code
  changes during the migration. Verified by branch diff: the only changes
  during the second migration are the regenerated files + the updated
  hand-written screen file.
- **SC-005**: Regenerating an unchanged UI-FBP spec produces byte-identical
  output (zero diff). Verified by `git diff --exit-code` after a
  re-generation pass.

## Assumptions

- The MVI pattern is applied at the per-screen/per-module granularity. Each
  UI-FBP module produces one State + Event + ViewModel + Screen pair. Modules
  with multiple screens are out-of-scope (today's UI-FBP generator already
  assumes one screen per module).
- Event-case naming derives mechanically from source-output port names
  (`Update<PortName>` for valued ports, `<PortName>` data-object for
  `Unit`-typed ports). Authors who want richer intent names can rename
  source-output ports — no separate intent-naming DSL is introduced in this
  feature.
- Each event case translates 1-to-1 to a source-output emission on the
  underlying controller. Multi-port emit (one event raising values on
  several source-output ports atomically) is out-of-scope; if needed it
  becomes a follow-up DSL extension.
- Default values for state properties are derived from the port's declared
  type using the same default-value table the existing `UIFBPStateGenerator`
  uses.
- The migration of DemoUI is the only hand-written-source change in this
  feature's primary scope. Updating the other DemoProjects is in-scope (P3)
  but each one is an independent atomic change.
- Under Design B (per the 2026-05-03 clarification), the generator change
  touches the State / Event / ViewModel / ControllerInterface / Runtime /
  SourceCodeNode / SinkCodeNode emissions. The Runtime Preview
  infrastructure outside the generator (PreviewRegistry, PreviewProvider
  registration, ModuleSessionFactory) stays as it is, and the
  PreviewProvider's emitted body is unchanged from feature 084. The
  hand-written `{Name}Screen` composable and any host-app root /
  navigation composable that wires the ViewModel to the Screen are NOT
  generated by UI-FBP — they remain entirely in the host module's
  hand-written code.
