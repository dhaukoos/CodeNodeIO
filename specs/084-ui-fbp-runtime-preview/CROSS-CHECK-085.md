# Feature 084 ↔ Feature 085 Cross-Check

**Date**: 2026-04-28
**Purpose**: Inventory of how feature 085 (universal-runtime collapse, shipped 2026-04-28) invalidates or simplifies parts of feature 084's planning artifacts, and what must be re-decided before resuming implementation.

This is the bridge document for the `/speckit.clarify` + `/speckit.plan` re-runs flagged in `spec.md`'s RESUMPTION NOTICE. It is **NOT** a rewrite of the artifacts — it identifies deltas; the targeted edits applied below in each artifact preserve the surviving design surface.

## What feature 085 changed (consumer-facing)

The thick per-module runtime stack is **gone across all modules** (StopWatch, Addresses, UserProfiles, EdgeArtFilter, WeatherForecast — and any new entity-style module produced by the Generate Module path):

| Before 085 | After 085 |
|---|---|
| `flow/{Module}Flow.kt` (hand-coded runtime) | **DELETED** |
| `controller/{Module}Controller.kt` (per-module class) | **DELETED** |
| `controller/{Module}ControllerAdapter.kt` | **DELETED** |
| `controller/{Module}ControllerInterface.kt` (declared start/stop/...) | **MODIFIED** — now `interface X : ModuleController` (control surface inherited) |
| (none) | **NEW** `controller/{Module}Runtime.kt` — factory `create{Module}Runtime(flowGraph): {Module}ControllerInterface` returning an anonymous `object : {Module}ControllerInterface, ModuleController by controller { ... }` over a `DynamicPipelineController` |
| (none) | **NEW** `userInterface/{Module}PreviewProvider.kt` (jvmMain) — generated by the new `PreviewProviderGenerator`, wired into all three generation paths |

## What this means for feature 084

The feature's spec body (US1–US4, most FRs, most SCs) is largely intact — UI-FBP modules still need a Runtime-Preview-loadable artifact set, still need a deployable artifact set, still need `.flow.kt` parse-and-merge, still need legacy `saved/` cleanup. **What's changed is the file shape that satisfies those needs**, and several items that were "this feature's responsibility" are now subsumed by the universal pipeline.

### Spec items obsoleted by 085

- **FR-005a** ("emit `Controller.kt` + `ControllerAdapter.kt` + `Flow.kt` runtime matching entity-module pattern"): GONE — those files no longer exist anywhere. The feature instead emits the post-085 single `controller/{Module}Runtime.kt` factory plus the modified `{Module}ControllerInterface.kt`.
- **SC-008** as drafted ("deployable artifacts structurally indistinguishable from an entity module's"): the *intent* (deployable parity) survives, but the *evidence* (matching `Controller`/`Adapter`/`Flow` files) is dead. Replace with: deployable evidence is a working `create{Module}Runtime(flowGraph)` factory + a `main()` consumer that wires the typed `{Module}ControllerInterface` into `{Module}ViewModel` (the template documented in feature 085's `quickstart.md` VS-D5).

### 084-scope work that is **already done** by feature 085

| 084 item | Status under 085 |
|---|---|
| Decision 3 (`{Module}PreviewProvider.kt` in jvmMain `userInterface/`) | **DONE** — `PreviewProviderGenerator` exists, is wired into `GenerationPath.UI_FBP` in `CodeGenerationRunner`, lands in `jvmMain` via `generatorToSourceSet`. |
| Decision 1's "thick stack reused via `RuntimeControllerGenerator`/`RuntimeControllerAdapterGenerator`/`RuntimeFlowGenerator`" | **OBSOLETE** — those generators are deleted. UI-FBP rides the universal runtime collapse just like entity modules. |
| `UIFBPSpecAdapter` (translate `UIFBPSpec` → entity-generator inputs) | **NOT NEEDED** — its only purpose was to feed the now-deleted entity generators. UI-FBP can call `ModuleRuntimeGenerator` and `RuntimeControllerInterfaceGenerator` directly with a `UIFBPSpec`-derived `GenerationConfig`. |
| `UIFBPControllerInterfaceGenerator` (NEW) | **PROBABLY NOT NEEDED** — `RuntimeControllerInterfaceGenerator` already emits the post-085 `interface X : ModuleController` shape with one `val y: StateFlow<T>` per observable port. UI-FBP's sink-input set should be expressible as `ObservableStateResolver`'s input set, in which case the existing generator suffices. **Verify before deciding to add a UI-FBP-specific variant.** |
| Hand-edit safety check (FR-016) for generated targets | **DONE in spirit** — `GenerationFileWriter.carriesGeneratorMarker` already implements this for the legacy-trio deletion path; the same pattern can be extended to the UI-FBP target paths if not already shared. |

### 084 contract revisions required

The `{Module}ControllerInterface` contract (`contracts/controller-interface.md`) needs:

1. **Inheritance**: now `interface X : ModuleController`, not a freestanding interface.
2. **Control methods**: `start/stop/pause/resume/reset/getStatus/setAttenuationDelay/setEmissionObserver/setValueObserver` come from `ModuleController` inheritance — generators MUST NOT redeclare them on the child interface.
3. **State-flow members**: now `override val y: StateFlow<T>` (not freestanding `val y`) because `ModuleController` declares the executionState surface; UI-FBP's port-flow members are *added* to the child interface and must be overridable in the Runtime factory's anonymous object.
4. **Two implementations**:
   - **Runtime Preview path**: reflection proxy in `ModuleSessionFactory` — works as before; the proxy now has explicit `getStatus` handling.
   - **Production-app path**: the post-085 `create{Module}Runtime(flowGraph)` factory's anonymous `object : {Module}ControllerInterface, ModuleController by controller { override val y = {Module}State.yFlow ... }`. **No more per-module `{Module}Controller` class.**

The verification document for the production-app path is now feature 085's `quickstart.md` VS-D5 ("Production-app integration template"). Feature 084's `quickstart.md` VS-C should reference it instead of re-deriving the pattern.

## Post-implementation reality check (added during T055)

Two assumptions in the original cross-check shifted during 084 implementation. Recording
the corrections so future readers see what actually landed.

1. **`UIFBPControllerInterfaceGenerator` (NEW) — said "PROBABLY NOT NEEDED"; actually was
   built inline.** `RuntimeControllerInterfaceGenerator` erases user IP types to `Any` via
   `KClass<*>.simpleName`, which would emit `val results: StateFlow<Any>` instead of
   `val results: StateFlow<CalculationResults?>`. To preserve typed IP types from
   `UIFBPSpec`, `UIFBPInterfaceGenerator.generateControllerInterface(...)` and
   `UIFBPInterfaceGenerator.generateRuntimeFactory(...)` emit the controller-interface
   and runtime-factory artifacts inline rather than delegating to the universal generators
   (research.md Decision 1, T016).

2. **`PreviewProviderGenerator` — extended in 084 land, not just reused.** A new optional
   `composableName: String?` parameter (T006) decouples the registry key (always the
   flow-graph prefix) from the in-lambda Composable function call (the user-authored
   function name from the qualifying UI source). This was needed for the post-082/083
   three-identifier model (Decision 2) — entity modules where prefix == composable name
   still get the existing behavior via the default.

Everything else in this document still holds — the universal `ModuleController`,
`DynamicPipelineController`, and the `getStatus()` proxy path are unchanged.
