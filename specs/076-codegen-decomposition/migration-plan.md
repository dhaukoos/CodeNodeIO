# Code Generation Migration Plan

**Feature**: 076-codegen-decomposition
**Date**: 2026-04-22
**Prerequisite**: dependency-analysis.md (completed in this feature)

## Overview

This document describes the sequence of future features needed to evolve CodeNodeIO's code generation system from monolithic orchestrator methods (`ModuleSaveService.saveModule()`, `saveEntityModule()`) into a configurable, flow-graph-based system where individual file generators are CodeNodes and generation paths are expressed as flow graphs.

Each step is scoped as an independent feature that can be specified, planned, and implemented on its own.

---

## Step 1: Folder Hierarchy Migration

**Scope**: Move existing generated module files from the current flat layout into the proposed subdirectory structure (flow/, controller/, viewmodel/, persistence/).

**Deliverables**:
- Update all generators to write files to the new subdirectory paths
- Update all import references in generated files to reflect new package paths
- Migrate existing generated modules (StopWatch, UserProfiles, Addresses, EdgeArtFilter, WeatherForecast) to the new layout
- Update `NodeDefinitionRegistry` scanning to find CodeNode files in the new locations
- Update `ModuleSessionFactory` and `DynamicPreviewDiscovery` to scan new paths

**Prerequisites**: Feature 076 (this feature — dependency analysis and proposed hierarchy)

**Enables**: All subsequent steps work with the new folder structure

**Risk**: Breaking change for existing modules. Requires careful migration of all demo project modules.

---

## Step 2: Module Scaffolding Extraction

**Scope**: Extract the module directory creation and Gradle file generation from `ModuleSaveService` into a standalone `ModuleScaffoldingGenerator` component.

**Deliverables**:
- New `ModuleScaffoldingGenerator` class with `generate(moduleName: String, outputDir: File): File` that creates the module directory structure and Gradle files
- Refactor `ModuleSaveService.saveModule()` to call `ModuleScaffoldingGenerator` first, then individual generators
- Refactor `ModuleSaveService.saveEntityModule()` similarly
- Unit tests for `ModuleScaffoldingGenerator` in isolation

**Prerequisites**: Step 1 (folder hierarchy — so scaffolding creates the right directories)

**Enables**: Step 3 (scaffolding becomes the root CodeNode in the generation flow graph)

**Key insight**: Module scaffolding is the root prerequisite. All other generators assume the directory structure exists. Extracting it makes this dependency explicit and testable.

---

## Step 3: Generator CodeNode Wrappers

**Scope**: Wrap each individual generator as a CodeNode with typed input and output ports, enabling them to participate in flow graphs.

**Deliverables**:
- Define a `GeneratorCodeNode` interface extending `CodeNodeDefinition` with `generate(input) → output` semantics
- Wrap each generator identified in the dependency analysis (15 potential CodeNodes from Section 6) as a CodeNode:
  - Input port: the spec/config data (FlowGraph, EntityModuleSpec, or UIFBPSpec)
  - Output port: the generated file content (String)
- Each wrapped CodeNode is independently testable
- CodeNodes are discoverable by the node palette

**Prerequisites**: Step 2 (scaffolding extracted — generators no longer assume directory creation)

**Enables**: Step 4 (CodeNodes can be composed into generation flow graphs)

**Design consideration**: The CodeNode wrappers should be thin — they delegate to the existing generator classes, adding only the port typing and CodeNodeDefinition interface. The existing generators remain unchanged.

---

## Step 4: Code Generation Flow Graphs

**Scope**: Create flow graphs that express generation paths as compositions of Generator CodeNodes, replacing the hardcoded orchestration in `ModuleSaveService`.

**Deliverables**:
- Create `GenerateModule.flow.kt` — a flow graph connecting Scaffolding → Flow + Controller + ViewModel + UI generators
- Create `GenerateRepository.flow.kt` — extends GenerateModule with Entity nodes, persistence, and entity UI generators
- Create `GenerateUIFBP.flow.kt` — connects Scaffolding → UIFBP generators + Controller + Flow generators
- Each flow graph is loadable in the graph editor and visually editable
- A `CodeGenerationRunner` that executes a generation flow graph with a given spec as input
- The file-tree checkbox selection from the Code Generator panel maps to enabling/disabling individual CodeNodes in the flow graph

**Prerequisites**: Step 3 (Generator CodeNodes exist and are composable)

**Enables**: Step 5 (Code Generator panel wires to these flow graphs)

**Key insight**: By expressing generation as a flow graph, users can customize which files are generated by editing the graph — adding, removing, or rewiring generators. The checkbox selection in the Code Generator panel is a simplified UI for the same concept.

---

## Step 5: Wire Code Generator Panel

**Scope**: Connect the Code Generator panel's file-tree selections and "Generate" button to the code generation flow graphs from Step 4.

**Deliverables**:
- The "Generate" button (currently disabled) becomes functional
- When pressed, the panel:
  1. Reads the selected generation path and primary input
  2. Loads the corresponding generation flow graph
  3. Enables/disables CodeNodes based on file-tree checkbox selections
  4. Executes the flow graph with the input spec
  5. Reports results (files created, errors) via status message
- Remove the old "Generate Module", "Generate UI-FBP" toolbar buttons (replaced by the panel)
- Remove or deprecate direct calls to `ModuleSaveService.saveModule()` and `saveEntityModule()`

**Prerequisites**: Step 4 (generation flow graphs exist and are executable)

**Enables**: Full configurable code generation — users can select exactly which files to generate

**Risk**: Must ensure backward compatibility — the panel must produce identical output to the current toolbar buttons for the same input.

---

## Step 6: User-Editable Generation Graphs (Future)

**Scope**: Allow users to open and edit the generation flow graphs in the graph editor, customizing the generation pipeline for their project.

**Deliverables**:
- Generation flow graphs are visible as a special category in the graph editor
- Users can add custom Generator CodeNodes (e.g., for project-specific boilerplate)
- Users can rewire the generation pipeline (e.g., skip UI generation, add custom persistence)
- Custom generation configurations are saved per-project

**Prerequisites**: Step 5 (panel wired, generation flow graphs work end-to-end)

**Enables**: Fully extensible code generation — the tool becomes a platform for custom code generation pipelines

---

## Dependency Chain

```
Feature 076 (this feature)
    └── Step 1: Folder Hierarchy Migration
        └── Step 2: Module Scaffolding Extraction
            └── Step 3: Generator CodeNode Wrappers
                └── Step 4: Code Generation Flow Graphs
                    └── Step 5: Wire Code Generator Panel
                        └── Step 6: User-Editable Generation Graphs (future)
```

Each step is strictly sequential — each depends on the deliverables of the prior step. However, each step is independently spec-able and implementable as its own feature branch.

| Step | Complexity | Estimated Tasks |
|------|-----------|-----------------|
| Step 1: Folder Migration | Medium | ~20 tasks (generator updates + module migration) |
| Step 2: Scaffolding Extraction | Small | ~10 tasks (extract + refactor + tests) |
| Step 3: CodeNode Wrappers | Medium | ~25 tasks (15 wrappers + tests + palette integration) |
| Step 4: Generation Flow Graphs | Large | ~30 tasks (3 flow graphs + runner + selection mapping) |
| Step 5: Wire Panel | Medium | ~15 tasks (button wiring + toolbar removal + backward compat) |
| Step 6: User-Editable | Large | ~25 tasks (editor integration + custom nodes + per-project save) |
