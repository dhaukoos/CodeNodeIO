# Implementation Plan: Architecture IP Types

**Branch**: `073-architecture-ip-types` | **Date**: 2026-04-14 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/073-architecture-ip-types/spec.md`

## Summary

Replace all `String::class` port types in `graphEditor/architecture.flow.kt` with domain-specific custom IP types. Types referencing existing domain classes use `typealias` declarations (e.g., `typealias NodeDescriptors = List<NodeTypeDefinition>`). Types without existing domain classes or with circular dependency constraints use `data class`. This involves: (1) extending IPTypeDiscovery to parse `typealias` syntax, (2) creating an `iptypes` Gradle module with 14 IP type files, and (3) updating all port declarations in architecture.flow.kt.

## IPTypeDiscovery Extension

### New regex pattern
```kotlin
private val typealiasPattern = Regex("""typealias\s+(\w+)\s*=\s*(.+)""")
```

### Modified parsing logic
After scanning for `@IPType` marker:
1. Try `dataClassPattern` first (existing behavior — backward-compatible)
2. If no `data class` found, try `typealiasPattern`
3. If typealias found, create `IPTypeFileMeta` with:
   - `typeName` from `@TypeName` header (or alias name from LHS)
   - `properties` = empty list
   - `className` = `packageName.aliasName`

### Test cases needed
- Parse typealias IP type file → correct IPTypeFileMeta
- Parse data class IP type file → unchanged behavior (backward compat)
- Parse file with both typealias and data class → data class takes precedence
- Parse file with `@IPType` but no data class or typealias → skip gracefully
