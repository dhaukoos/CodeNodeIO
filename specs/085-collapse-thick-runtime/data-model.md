# Phase 1 Data Model: Universal Runtime Collapse

**Date**: 2026-04-27
**Feature**: [spec.md](./spec.md) · **Plan**: [plan.md](./plan.md) · **Research**: [research.md](./research.md)

This feature introduces no new persistence and no new domain model in the user/application sense. What it does change is the **shape of the generator's outputs** and the **contract surface** between generated modules and their consumers (both the GraphEditor's Runtime Preview and KMPMobileApp's production deployment). The entities described here are the generator inputs/outputs, the new per-module file's structure, and the `getStatus()` data shape carried through the universal runtime.

---

## 1. ModuleSpec — generator input (existing, unchanged)

The generator orchestrator (`ModuleGenerator`) consumes a per-module specification that already exists today. This feature does not modify its shape. The orchestrator extracts what it needs to populate `ModuleRuntimeGenerator`'s input and updates which downstream generators it invokes.

Key fields (pre-existing) that this feature consumes:
- `moduleName: String` — e.g., "StopWatch"
- `packageName: String` — e.g., "io.codenode.stopwatch"
- `nodes: List<NodeSpec>` — each with a `name` and a fully-qualified `CodeNodeDefinition` reference
- `controllerInterfaceMembers` — the per-port `StateFlow` getter set used by `RuntimeControllerInterfaceGenerator`
- `stateObjectFqcn: String` — e.g., "io.codenode.stopwatch.viewmodel.StopWatchState" (target of the `onReset` callback and source of the `{x}Flow` fields the factory delegates to)

## 2. ModuleRuntimeSpec (NEW) — input to the new generator

Lives in `flowGraph-generate/src/commonMain/kotlin/io/codenode/flowgraphgenerate/generator/ModuleRuntimeSpec.kt`.

```kotlin
data class ModuleRuntimeSpec(
    val moduleName: String,                  // e.g., "StopWatch"
    val packageName: String,                 // e.g., "io.codenode.stopwatch"
    val controllerInterfaceFqcn: String,     // e.g., "io.codenode.stopwatch.controller.StopWatchControllerInterface"
    val stateObjectFqcn: String,             // e.g., "io.codenode.stopwatch.viewmodel.StopWatchState"
    val nodeRegistryEntries: List<NodeRegistryEntry>,
    val typedStateFlows: List<TypedStateFlow>
)

data class NodeRegistryEntry(
    val nodeName: String,                    // The string used in the FlowGraph DSL (e.g., "TimerEmitter")
    val codeNodeDefinitionFqcn: String       // e.g., "io.codenode.stopwatch.nodes.TimerEmitterCodeNode"
)

data class TypedStateFlow(
    val propertyName: String,                // e.g., "elapsedSeconds"
    val typeName: String,                    // e.g., "Int" — verbatim Kotlin type rendering
    val isNullable: Boolean = false,         // e.g., true for "CalculationResults?"
    val stateFieldName: String               // e.g., "elapsedSecondsFlow" — the field on the State object
)
```

**Validation rules**:
- `moduleName` MUST be a non-empty Kotlin identifier in PascalCase.
- `packageName` MUST be a valid Kotlin package path.
- `nodeRegistryEntries` MAY be empty (a degenerate module with no CodeNodes — accepted but emits a `lookup` that returns `null` for all names).
- Every `TypedStateFlow.stateFieldName` MUST exist as a public `val` of type `StateFlow<T>` on the resolved `stateObjectFqcn`. The generator does NOT verify this at emission time (it's a string template), but the generated file's compilation will fail loudly if the contract is broken.

**Source**: derived by `ModuleGenerator` from the existing module spec; no new user-facing input format.

---

## 3. ModuleRuntimeGenerator (NEW) — generator class

Lives in `flowGraph-generate/src/commonMain/kotlin/io/codenode/flowgraphgenerate/generator/ModuleRuntimeGenerator.kt`. Pure, single-method, KMP-clean.

```kotlin
class ModuleRuntimeGenerator {
    fun generate(spec: ModuleRuntimeSpec): String
}
```

**Output shape**: a single Kotlin source file whose body matches the template in research Decision 5. The generated file is placed at `{moduleRoot}/src/commonMain/kotlin/{packagePath}/{ModuleName}Runtime.kt` (where `{packagePath}` is `packageName` with `.` → `/`). For StopWatch (packageName `io.codenode.stopwatch`), this is `StopWatch/src/commonMain/kotlin/io/codenode/stopwatch/StopWatchRuntime.kt`.

**Idempotency**: identical input produces byte-identical output. The generator is pure; no I/O.

## 4. Generated `{Module}Runtime.kt` (NEW per-module file) — output structure

Reference template (StopWatch):

```kotlin
/*
 * StopWatchRuntime — Universal-runtime factory for the StopWatch module
 * Generated by CodeNodeIO ModuleRuntimeGenerator
 * License: Apache 2.0
 */

package io.codenode.stopwatch

import io.codenode.fbpdsl.model.FlowGraph
import io.codenode.fbpdsl.runtime.CodeNodeDefinition
import io.codenode.fbpdsl.runtime.DynamicPipelineController
import io.codenode.fbpdsl.runtime.ModuleController
import io.codenode.stopwatch.controller.StopWatchControllerInterface
import io.codenode.stopwatch.nodes.DisplayReceiverCodeNode
import io.codenode.stopwatch.nodes.TimeIncrementerCodeNode
import io.codenode.stopwatch.nodes.TimerEmitterCodeNode
import io.codenode.stopwatch.viewmodel.StopWatchState

object StopWatchNodeRegistry {
    fun lookup(name: String): CodeNodeDefinition? = when (name) {
        "TimerEmitter"     -> TimerEmitterCodeNode
        "TimeIncrementer"  -> TimeIncrementerCodeNode
        "DisplayReceiver"  -> DisplayReceiverCodeNode
        else               -> null
    }
}

fun createStopWatchRuntime(flowGraph: FlowGraph): StopWatchControllerInterface {
    val controller = DynamicPipelineController(
        flowGraphProvider = { flowGraph },
        lookup = StopWatchNodeRegistry::lookup,
        onReset = StopWatchState::reset
    )
    return object : StopWatchControllerInterface, ModuleController by controller {
        override val elapsedSeconds = StopWatchState.elapsedSecondsFlow
        override val elapsedMinutes = StopWatchState.elapsedMinutesFlow
        override val seconds = StopWatchState.secondsFlow
        override val minutes = StopWatchState.minutesFlow
    }
}
```

**File-level invariants**:
- The file lives at the module's package root (not a subpackage).
- Filename always `{ModuleName}Runtime.kt`.
- Contains exactly one `object {ModuleName}NodeRegistry` and exactly one top-level function `create{ModuleName}Runtime(flowGraph: FlowGraph): {ModuleName}ControllerInterface`.
- Header carries the `Generated by CodeNodeIO ModuleRuntimeGenerator` marker comment (used by the safety check in Decision 9 of research.md).
- The `object` expression in the factory uses `ModuleController by controller` Kotlin interface delegation to forward all inherited control methods (`start`, `stop`, `pause`, `resume`, `reset`, `executionState`, `setAttenuationDelay`, `setEmissionObserver`, `setValueObserver`, `getStatus`, `nodeDefinitionLookup`) to the underlying `DynamicPipelineController`.
- Per-port `StateFlow` getters are emitted as explicit `override val xxx = {Module}State.xxxFlow` (cannot use Kotlin delegation because they come from a different source than the controller).

### 4a. DAO-bearing modules (Addresses, UserProfiles)

The factory function shape is **identical** for entity modules with persistence — the factory still returns `{Module}ControllerInterface` and the registry/factory body is unchanged. The DAO is NOT a parameter to the factory; it stays at the consumer's call site, passed to the ViewModel constructor:

```kotlin
// Generated AddressesRuntime.kt — identical pattern to StopWatch
fun createAddressesRuntime(flowGraph: FlowGraph): AddressesControllerInterface {
    val controller = DynamicPipelineController(
        flowGraphProvider = { flowGraph },
        lookup = AddressesNodeRegistry::lookup,
        onReset = AddressesState::reset
    )
    return object : AddressesControllerInterface, ModuleController by controller {
        override val save = AddressesState.saveFlow
        override val update = AddressesState.updateFlow
        override val remove = AddressesState.removeFlow
        override val result = AddressesState.resultFlow
        override val error = AddressesState.errorFlow
    }
}

// At the consumer (KMPMobileApp/App.kt):
val controller = createAddressesRuntime(addressesFlowGraph)        // returns AddressesControllerInterface
val viewModel = AddressesViewModel(controller, AddressesPersistence.dao)  // ViewModel takes (Interface, Dao)
```

The two-arg ViewModel constructor (`AddressesViewModel(ControllerInterface, AddressDao)`, `UserProfilesViewModel(ControllerInterface, UserProfileDao)`) is preserved by `RuntimeViewModelGenerator` (out of scope for this feature). The DAO comes from the persistence module's static accessor at the consumer's call site, exactly as today. The factory function does **not** know about DAOs — keeping it generic is what allows a single `ModuleRuntimeGenerator` template to serve every module type.

## 5. ModuleGenerator orchestration changes

The existing orchestrator (`flowGraph-generate/src/commonMain/.../generator/ModuleGenerator.kt`) is modified at three points:

| Change | What |
|---|---|
| **Stop invoking** | `RuntimeControllerGenerator`, `RuntimeControllerAdapterGenerator`, `RuntimeFlowGenerator`. Their call sites are removed. |
| **Start invoking** | `ModuleRuntimeGenerator` once per module, with a `ModuleRuntimeSpec` derived from the existing per-module spec. |
| **Schedule deletions** | For each existing module being regenerated, schedule deletion of `controller/{Module}Controller.kt`, `controller/{Module}ControllerAdapter.kt`, `flow/{Module}Flow.kt`. Each deletion goes through the safety check (refuse if file lacks the `Generated by CodeNodeIO` marker per FR-013). |

**No new orchestration entry point**; the change is internal to `ModuleGenerator`'s existing per-module rendering loop.

## 6. RuntimeControllerInterfaceGenerator change

The existing generator (`flowGraph-generate/src/commonMain/.../generator/RuntimeControllerInterfaceGenerator.kt`) is modified to:
1. Add `import io.codenode.fbpdsl.runtime.ModuleController` to the emitted import block.
2. Change the emitted interface declaration from `interface XControllerInterface { ... }` to `interface XControllerInterface : ModuleController { ... }`.
3. Remove the explicit `executionState`, `start`, `stop`, `pause`, `resume`, `reset` declarations from the interface body — these are now inherited from `ModuleController`. (Keep the per-port `val xxx: StateFlow<T>` declarations.)

The unit test for this generator (`RuntimeControllerInterfaceGeneratorTest`) gets one additional assertion: the emitted source contains the `: ModuleController` superinterface clause, and does NOT redeclare the inherited members.

## 7. fbpDsl changes — `getStatus()` addition

### `ModuleController.kt`

```kotlin
interface ModuleController {
    val executionState: StateFlow<ExecutionState>
    fun start(): FlowGraph
    fun stop(): FlowGraph
    fun pause(): FlowGraph
    fun resume(): FlowGraph
    fun reset(): FlowGraph
    fun setAttenuationDelay(ms: Long?)
    fun setEmissionObserver(observer: ((String, Int) -> Unit)?)
    fun setValueObserver(observer: ((String, Int, Any?) -> Unit)?)

    /** NEW: returns a snapshot of per-node execution state for the current pipeline. */
    fun getStatus(): FlowExecutionStatus

    var nodeDefinitionLookup: ((String) -> CodeNodeDefinition?)?
        get() = null
        set(_) {}
}
```

### `DynamicPipeline.kt`

Add a method that wraps the internal `RootControlNode` (which `DynamicPipeline` already constructs):

```kotlin
fun getStatus(): FlowExecutionStatus = rootControlNode.getStatus()
```

Where `rootControlNode` is the existing internal `RootControlNode` instance — currently used implicitly during start/stop. This requires retaining a reference to it as a class field (~1 line change in the existing pipeline construction).

### `DynamicPipelineController.kt`

```kotlin
override fun getStatus(): FlowExecutionStatus =
    pipeline?.getStatus() ?: FlowExecutionStatus(overallState = ExecutionState.IDLE, perNode = emptyMap())
```

(Exact `FlowExecutionStatus` empty-state shape depends on its existing constructor; the test-first task confirms.)

### Test

`DynamicPipelineControllerTest.kt` gains a test asserting:
- Before `start()`, `getStatus()` returns IDLE with empty per-node map.
- After `start()`, `getStatus()` reflects RUNNING and lists every node with a state.
- After `pause()`, `getStatus()` reflects PAUSED.
- After `stop()`, `getStatus()` reverts to IDLE.

## 8. ModuleSessionFactory proxy change (one line)

`flowGraph-execute/src/jvmMain/kotlin/io/codenode/flowgraphexecute/ModuleSessionFactory.kt:108-150` (`createControllerProxy`) gains one explicit case in its `when` branch on `method.name`:

```kotlin
"getStatus" -> controller.getStatus()
```

Without this, calls to `getStatus()` through the GraphEditor's reflection proxy would fall into the `else null` branch and return null (causing NPE in any consumer that uses the typed interface inside Runtime Preview). KMPMobileApp doesn't use the proxy path, so this is a defensive add — but the proxy is the typed interface's only Runtime Preview implementation, and `getStatus` is now part of that surface.

## 9. State transitions / lifecycle (per-module regeneration loop)

The orchestrator's per-module loop has the following decision shape:

```
For each target file under {moduleRoot}:

  Is the file in the SCHEDULED-DELETE set (Controller.kt, Adapter.kt, Flow.kt)?
    YES → Read first ~5 lines.
            Contains "Generated by CodeNodeIO" marker?
              YES → DELETE; record kind=DELETED.
              NO  → SKIP; record kind=SKIPPED_CONFLICT with reason "marker absent".
    NO  → Continue.

  Is the file the NEW {Module}Runtime.kt?
    YES → File exists?
            NO  → WRITE; record kind=CREATED.
            YES → Content matches would-emit?
                    YES → record kind=UNCHANGED.
                    NO  → CARRIES the Generated marker?
                            YES → WRITE; record kind=UPDATED.
                            NO  → SKIP; record kind=SKIPPED_CONFLICT.

  Is the file the MODIFIED {Module}ControllerInterface.kt?
    Same logic as Runtime.kt.

  Other files (ControllerInterface unmodified, State, ViewModel, nodes, .flow.kt, PreviewProvider) → UNCHANGED.
```

The structured summary (per FR-013) collects every `FileChange { relativePath, kind, reason? }` entry per module and surfaces it to the user.

## 10. Updated `{Module}ControllerInterface.kt` shape

After modification by `RuntimeControllerInterfaceGenerator`, every `{Module}ControllerInterface` looks like this (StopWatch as reference):

```kotlin
/*
 * StopWatchControllerInterface
 * Generated by CodeNodeIO RuntimeControllerInterfaceGenerator
 * License: Apache 2.0
 */

package io.codenode.stopwatch.controller

import io.codenode.fbpdsl.runtime.ModuleController
import kotlinx.coroutines.flow.StateFlow

interface StopWatchControllerInterface : ModuleController {
    val elapsedSeconds: StateFlow<Int>
    val elapsedMinutes: StateFlow<Int>
    val seconds: StateFlow<Int>
    val minutes: StateFlow<Int>
}
```

Note the absence of explicit `start()`, `stop()`, etc. — they're inherited. The interface now contains only its truly module-specific members (the typed state-flow getters).

## 11. What this feature does NOT model

- No new persistence; no database schemas; no migration scripts.
- No changes to `FlowGraph`, `CodeNode`, `Node`, or any `fbpDsl` model classes other than `ModuleController` (interface) and `DynamicPipelineController` / `DynamicPipeline` (added `getStatus`).
- No changes to `PreviewRegistry`, `DynamicPreviewDiscovery`, `RuntimeRegistry`, or `RootControlNode` (all reused as-is).
- No changes to the GraphEditor UI surface (`graphEditor/`).
- No changes to the persistence module (`persistence/`).
- No new IP types; existing IP types continue to be discovered by `IPTypeRegistry`.
- No new generators beyond `ModuleRuntimeGenerator`. The three eliminated generators have no replacement — their work is subsumed by `DynamicPipelineBuilder` (already in fbpDsl).
- The `setNodeState`/`setNodeConfig` capabilities present on the deleted per-module Controllers are NOT carried to the universal runtime (no consumer; deferred to a future feature if needed).
- The `bindToLifecycle(Lifecycle)` capability is NOT carried (Android-specific, no consumer; UI layer manages lifecycle directly).
