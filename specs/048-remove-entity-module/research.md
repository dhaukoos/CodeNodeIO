# Research: Remove Repository Module

**Feature**: 048-remove-entity-module | **Date**: 2026-03-11

## Research Tasks

### 1. How are entity module artifacts created?

**Decision**: The creation flow is fully understood. `ModuleSaveService.saveEntityModule()` orchestrates all artifact creation.

**Findings**:
- **Custom Nodes**: 3 `CustomNodeDefinition` objects (CUD, Repository, Display) added via `FileCustomNodeRepository.add()`, persisted to `~/.codenode/custom-nodes.json`. Each has `sourceIPTypeId` linking to the IP Type.
- **Module Directory**: Created at `{projectRoot}/{PluralName}/` with subdirectories for source sets, generated code, processing logic, and UI.
- **Persistence Files**: `{Entity}Entity.kt`, `{Entity}Dao.kt`, `{Entity}Repository.kt` written to `persistence/src/commonMain/kotlin/io/codenode/persistence/`.
- **AppDatabase.kt**: Regenerated by scanning all `*Entity.kt` files in persistence dir, collecting entity info, and calling `repositoryCodeGenerator.generateDatabase()`.
- **Gradle entries**: `include(":ModuleName")` added to `settings.gradle.kts`; `implementation(project(":ModuleName"))` added to `graphEditor/build.gradle.kts`.

**Rationale**: Removal is the exact inverse — delete/remove each artifact category.

### 2. How to remove custom nodes by IP Type?

**Decision**: Use `FileCustomNodeRepository.remove(id)` for each node matching `sourceIPTypeId`.

**Findings**:
- `FileCustomNodeRepository` exposes `remove(id: String)` which removes by node ID and saves.
- `getAll()` returns all nodes; filter by `sourceIPTypeId` to find the 3 entity module nodes.
- Each entity module creates exactly 3 nodes: one with `isCudSource=true`, one with `isRepository=true`, one with `isDisplay=true`.

**Alternatives considered**:
- Adding a `removeBySourceIPTypeId()` method — unnecessary, filtering + individual remove is sufficient and simpler.

### 3. How to regenerate AppDatabase.kt after entity removal?

**Decision**: Reuse the existing scanning pattern from `ModuleSaveService.regenerateAppDatabase()`.

**Findings**:
- After deleting persistence files, scan remaining `*Entity.kt` files in persistence dir.
- Collect `EntityInfo(entityName, tableName, daoName)` for each.
- Call `repositoryCodeGenerator.generateDatabase(entityInfos, packageName)`.
- Write the result to `AppDatabase.kt`.
- The existing `pluralize()` function handles table name derivation.

**Rationale**: This pattern is already proven and handles the multi-entity case correctly.

### 4. How to remove Gradle entries?

**Decision**: Read file content, filter out matching lines, write back.

**Findings**:
- `settings.gradle.kts`: Remove lines matching `include(":ModuleName")`.
- `graphEditor/build.gradle.kts`: Remove lines matching `implementation(project(":ModuleName"))`.
- Both are simple text operations — read lines, filter, write back.

**Alternatives considered**:
- Regex-based removal — overkill for single-line entries with known exact format.

### 5. How to handle the confirmation dialog?

**Decision**: Use Compose Material `AlertDialog` in `Main.kt`, gated by a `showRemoveConfirmDialog` state variable.

**Findings**:
- Compose Desktop supports `AlertDialog` from `androidx.compose.material`.
- Pattern: `var showDialog by remember { mutableStateOf(false) }` → button sets true → dialog confirms/cancels → confirmed triggers removal.
- Follows existing Compose Material patterns used elsewhere in the app.

### 6. How to update UI state after removal?

**Decision**: Reload `customNodes` from repository after removal, which triggers recomposition.

**Findings**:
- `customNodes` is a `mutableStateOf` in Main.kt.
- After removal, `customNodes = customNodeRepository.getAll()` refreshes the list.
- The `moduleExists` check (`customNodes.any { ... }`) will evaluate to false, showing the "Create" button.
- No additional state management needed — Compose reactivity handles it.
